阶段 1 深入拆解 · 本地输入编排

REPL 里的这次输入，究竟如何被整理成一次可执行的 turn？

这一页只讨论阶段 1 的边界之内发生的事：用户在 REPL 中键入文本、切换输入模式、插入引用、携带粘贴内容， 最终由 handlePromptSubmit 规范化成一份可以继续分流的提交请求。到这一步为止，系统仍主要停留在本地 UI 层， 还没有真正进入 query.ts 的 API 采样主循环。

阶段 1 的核心结论

PromptInput 不是一个单纯的文本框，而是一次 turn 的前置编排器。它需要同时维护输入字符串、模式、 引用占位符、图片或文本粘贴内容、建议状态、历史状态，以及当前会话上下文。提交动作真正做的，不是“把文本发给模型”， 而是先把这些零散状态收束成一次受控的本地提交。

输入还没进入模型

这一层的职责是采集和规范化，而不是采样。真正进入模型要到后面的 processUserInput 和 query。

模式先于路由

prompt、bash、/slash 会决定后续分支，但判断发生在本地提交前后，不在 API 层。

引用不是纯展示

用户看到的是占位符，提交前会经过 parseReferences 和文本展开，未被引用的图片也会被过滤掉。

提交不等于立刻执行

如果当前 turn 还在跑，queryGuard 会把这次输入送进队列，等待统一调度，而不是直接并发执行。

阶段边界

先把阶段 1 的“到哪为止”划清楚。很多读源码时的混乱，来自把本地输入编排和后续模型循环揉成了一层。

阶段 1 内

本地交互与提交准备

  • PromptInput 维护 input、mode、pastedContents、建议和历史状态。
  • 按下 Enter 后，onSubmit 会做建议拦截、空输入判断、图片特判、代理路由。
  • handlePromptSubmit 会过滤图片、展开文本引用、识别即时命令、决定排队还是立即执行。
阶段 1 边界点

进入统一执行入口

  • 直接执行路径会把输入包装成一个 QueuedCommand，再交给 executeUserInput。
  • 排队路径会把同样的输入放进 messageQueueManager，稍后由队列处理器消费。
  • 到这里为止，提交对象已经成型，但还没进入真正的模型采样循环。
阶段 2 起

输入开始进入真正分流

  • processUserInput 开始根据 mode 和前缀，进入 prompt、bash 或 slash command 分支。
  • processTextPrompt、processBashCommand、processSlashCommand 开始产出真正的消息对象。
  • 随后才会进入 query.ts 的消息规范化与流式采样主线。

输入生命周期

把一次普通提交拆成 6 个连续动作，更容易理解为什么“文本框输入”在这里其实是一条局部状态机。

用户在 PromptInput 中输入内容

PromptInput.tsx 内部维护的不只是字符串。它还同时关心建议列表、历史回放、stash、引用 pill、 图片引用位置、底部状态栏和模式指示器。此时看到的内容是“可编辑表现层”，并不等于最终提交给后端的原始内容。

输入模式在本地即时变化

inputModes.ts 会把以 ! 开头的输入识别成 bash 模式，普通文本维持 prompt。这一步只是在 UI 层决定后续解释方式，并没有马上触发 bash 工具或 slash command。

引用和粘贴内容以占位符存在

文本粘贴和图片粘贴不会直接内联到输入字符串里，而是通过类似 [Image #N]、文本引用占位符的形式留在输入中， 真正内容放在 pastedContents。这让用户可以删掉 pill，也让提交前有机会重新过滤和展开。

按下 Enter 先经过 PromptInput 自己的守门逻辑

PromptInput 会先看建议列表是否仍在显示、空输入是否只是想接受建议、是否在给 teammate 发送消息、 是否只提交图片、是否需要优先处理本地交互。也就是说，Enter 先归 UI 自己解释，再决定是否真正提交。

handlePromptSubmit 规范化输入

这里会做 5 个关键动作：过滤未引用图片、展开文本引用、拦截 exit / /exit、 检查是否属于即时本地 slash command、以及在当前 turn 活跃时决定是否入队。到了这里，系统才形成一份稳定的“可执行提交”。

统一包装成 queued command 语义

无论是立即执行还是先排队，Claude Code 都倾向于把输入包装成统一的 QueuedCommand 语义， 这样后续 executeUserInput 和队列消费路径可以复用同一套处理逻辑，避免 prompt、图片和引用在两条路径上行为不一致。

四类关键状态

如果只记住一个源码阅读框架，就记住这四类状态分别负责什么，它们一起构成阶段 1 的主视角。

input

用户正在编辑的可见字符串。它可能带有 !、/ 和引用占位符，但还不是最终消息体。

mode

prompt 或 bash。它决定后续怎样解释输入，但 slash command 仍然靠前缀 / 再判断。

pastedContents

真实的图片或文本粘贴内容仓库。占位符在字符串里，实体在这里，提交时会做引用一致性检查。

queuedCommands

等待执行的提交快照。当前 turn 忙时，新输入不会消失，而是先落到统一命令队列里。

三条分流路径

阶段 1 不会执行所有逻辑，但它必须先把输入预分成可理解的路径。真正的执行分支会在后面展开。

普通 prompt

默认路径。输入以自然语言文本为主，可能附带引用、图片和 IDE 选择内容。后续在 processUserInput 中进入 processTextPrompt，被包装成真正的 user message。

  • 没有 ! 模式前缀。
  • 没有以 / 起始的 slash command 语义。
  • 允许附件、图片和文本引用一起进入后续上下文。

bash 模式

由 ! 前缀触发，本地先把模式识别成 bash。提交后的文本值会去掉前缀字符， 后续由 processBashCommand 继续处理，不是 UI 层直接执行 shell。

  • getModeFromInput 用前缀字符判定。
  • prependModeCharacterToInput 负责历史与展示的一致性。
  • 真正的工具调用仍然在更后面的执行层发生。

slash command

输入以 / 开头时，handlePromptSubmit 会先尝试识别是否为即时本地命令。 如果是 local-jsx 且满足条件，甚至可以在 query 活跃时直接弹本地 UI；否则后续进入 processSlashCommand 路径。

  • 本地即时命令典型如配置和诊断类面板。
  • skipSlashCommands 可阻止远端桥接消息误触发本地 slash 行为。
  • 未知 slash 输入可回退成普通文本语义，而不是强制报错。

提交瞬间的规范化动作

真正让阶段 1 变得“像一个系统”的，就是这几个细节。它们看起来琐碎，但共同决定输入是否稳定可重放。

1. 未引用图片会被丢弃

handlePromptSubmit 会先用 parseReferences(input) 找出仍然保留在文本里的引用编号。 图片如果已经没有对应占位符，就不会继续跟着这次提交走。这意味着“删掉 pill”在语义上是真删除。

2. 文本引用会先展开

文本粘贴内容不是把编号发给模型，而是通过 expandPastedTextRefs 在提交前替换成真实文本。 队列路径和立即执行路径都会共用这一步，保证后续行为一致。

3. exit 有本地短路逻辑

exit、quit、:q 之类输入不会按普通 prompt 继续流转，而是优先被改写成 /exit 或直接本地退出逻辑。这是本地会话控制，不属于模型推理的一部分。

4. query 活跃时先排队

如果 queryGuard.isActive 为真，或者外部 loading 正在进行，本次提交会被 enqueue 送入统一队列。用户已经“提交成功”，但系统会延迟执行，避免多个 turn 并发污染消息链。
