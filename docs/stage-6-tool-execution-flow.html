并发批次策略：partitionToolCalls

编排层不是把所有 tool_use 一股脑丢出去。它先分区，再按安全级别决定能否并行。

并行批（Concurrent）

isConcurrencySafe = true
  • 读文件（Read、Glob、Grep）
  • 截图、WebSearch 等只读操作
  • 最多 10 个同时执行
  • 上下文修改器在批次结束后统一应用

串行批（Serial）

isConcurrencySafe = false
  • 写文件（Edit、Write）
  • 执行命令（Bash）
  • 每个工具单独执行、立即更新上下文
  • 确保后续工具看到前一个的副作用

分区算法

partitionToolCalls()
  • 遍历 tool_use 列表
  • 连续的 safe 工具归入同一批
  • 遇到 unsafe 工具立即截断，单独成批
  • 结果是交替的 concurrent / serial 批次序列

权限决策的五层架构

权限判断不是一个 if-else，而是五层逐级 fallback 的决策链。核心不变量：Hook 的 allow 不能越过 Rule 的 deny。

Allow（允许执行）

  • 匹配到 allow 规则（settings / project / CLI）
  • bypassPermissions 模式下所有工具通过
  • acceptEdits 模式下安全编辑操作通过
  • auto 模式下分类器评估为安全
  • 用户在弹窗中确认允许

允许后可以携带 updatedInput——允许的同时还能修改输入参数。

Ask（需要确认）

  • 匹配到 ask 规则，触发用户弹窗
  • default 模式下大多数写操作
  • auto 模式下分类器不确定时回退
  • 工具自身的 checkPermissions 返回 ask
  • 可附带 suggestions 帮助用户理解选项

弹窗可展示 contentBlocks，让用户看到即将执行的具体内容。

Deny（拒绝执行）

  • 匹配到 deny 规则，直接拒绝
  • dontAsk 模式下 ask 自动降级为 deny
  • auto 模式下分类器判定为危险
  • 安全检查（敏感路径访问）失败
  • 连续拒绝次数超过阈值时回退交互

拒绝时返回错误 tool_result，模型会看到拒绝原因并可能调整策略。

安全不变量：PreToolUse hook 的 allow 决策不能越过 deny/ask 规则。 即使 hook 说允许，系统仍然会检查规则层。这确保了用户设定的安全边界不会被 hook 绕过。

决策来源追踪：每个 PermissionDecision 都携带 decisionReason， 记录是 rule / mode / hook / classifier / safetyCheck 哪一层做出的决策，便于调试和审计。

上下文修改器模式：工具执行后可以返回 contextModifier， 比如 Bash 工具改变了 cwd，会通过修改器更新 ToolUseContext。 并行批中这些修改器会排队等待，串行批中会立即生效。

教学要点

给别人讲工具执行时，优先讲清这 4 个事实。

工具不是 REPL 直接调用的函数，而是模型在采样过程中发出的 tool_use 指令， 经过编排 → 校验 → 权限 → 执行 → 后处理的完整管线才落地。

并发是有条件的。只读工具可以并行（最多 10 个），写操作必须串行， 并且上下文修改器要在正确的时机应用。

权限是多层的，不是一个开关。规则、模式、Hook、分类器、用户确认五层逐级 fallback， 而且 Hook 的 allow 不能越过规则的 deny——这是安全不变量。

工具结果不是终点。tool_result 回灌到消息链后，query 继续下一轮采样。 一次用户提交可能经历多轮 tool_use → tool_result 闭环。
