Claude Code 学习页 · 用户输入到输出结果的完整数据流

从用户发起 prompt / token，到终端渲染结果，中间到底发生了什么？

这个页面把当前仓库里的核心运行主线压成一条可视化教学链路。重点不是列所有模块，而是帮助开发者建立一张稳定心智图： 用户输入进入 REPL 之后，如何被分流、入队、交给 Query 主循环，什么时候触发工具执行，哪些内容会被写入会话日志，以及最终为什么会在终端看到流式结果。

核心观察
这是一条可恢复、可持久化、可插工具的长链路，不是一次简单的 API 调用。

从 `REPL.tsx` 的输入提交，到 `handlePromptSubmit` 的预处理，再到 `query.ts` 的循环采样和 `toolOrchestration.ts` 的工具执行，最后通过 `sessionStorage.ts` 与终端消息列表汇合，这条链路同时在处理控制流、消息流和 token 预算。

输入不会直接发 API

先经过输入模式识别、引用展开、队列调度和 query guard。

Query 是核心引擎

真正负责消息规范化、采样、工具回路、compact 与 stop hooks。

工具执行是回路

模型返回 `tool_use`，本地执行后再以 `tool_result` 回灌给模型。

输出不只显示在屏幕上

会话日志、恢复信息、工具结果存储和 REPL 渲染会同步更新。

    把源码读成一条时间线

    先沿主干走，再按需向队列、工具、持久化这些侧边能力展开。

    提交前

    用户在 `PromptInput` 中输入文本，REPL 仍然处于本地交互阶段，尚未进入模型采样。

    提交瞬间

    `handlePromptSubmit` 整理消息、引用、模式、队列 reservation，并决定是直接执行还是先入队。

    执行阶段

    `query.ts` 进入循环：准备 API 消息、流式接收 assistant 事件、必要时调用工具，再把 `tool_result` 写回消息链。

    收尾阶段

    `sessionStorage.ts` 落盘 transcript，REPL 列表更新，恢复机制和统计系统也获得这轮会话状态。

    教学提示

    给别人讲这条链路时，优先讲清这 4 个事实。

    `REPL` 是交互壳，不是核心推理引擎。真正驱动输入变输出的是 `handlePromptSubmit -> query -> tool orchestration`。
    工具不是 REPL 直接执行的“按钮”，而是模型在采样过程中发出的 `tool_use` 指令，再由本地工具系统落地。
    持久化不是附属功能。`sessionStorage.ts` 直接影响恢复、继续会话、统计、日志重放和调试能力。
    token 不只存在于输入和输出两头。附件、工具结果、compact 和 continuation 都会改变整轮成本结构。
